Peter, new idea
It's to do with indexes. 

Assuming that a developer always codes using the Index.N syntax, tracking down where an index is used is easy. But, there are other ways....

Send Find of oMy_DD EQ 3

vFind MyTable.File_Number 3 GT
and even 
Find EQ MyTable.MyField

where it uses the associated field index as the index to use, which could be 3.
So many times I've wanted to change an index but the thought of tracking down all the instances where that index is used is too much. The challenge would be to see what might be possible here.

Of course, sometimes you just cannot do everything. For example:

Procedure FindMyRecord Integer hTable Integer iIndex
    vFind hTable iIndex EQ
End_Procedure

..ok, silly example but you get the point that the index is passed as an arg, so you just couldn't know where it was called using the index number you are interested in. 

And of course, you have things like First_Record, Next_Record, Locate_Next etc, so trying to track down all the lines that are doing a find and therefore will require an index would be quite tricky. 

Again, it's more of a report than a code change. But, incredibly challenging to get as many "hits" as possible
